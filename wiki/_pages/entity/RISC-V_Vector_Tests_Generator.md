# RISC-V Vector Tests Generator

The RISC-V Vector Tests Generator, hosted under the Chips Alliance organization, is an open-source tool for automatically generating unit tests for the RISC-V Vector Extension (RVV) version 1.0. It is designed to be simple and easy to use, following the pattern of the well-known riscv-tests project. The generator produces self-verifying test binaries that can be used either for bare-metal testing (machine-mode) or user-mode execution via riscv-pk. It supports both RV32 and RV64 base architectures, tests element widths (SEW) from e8 to e64, and vector length multipliers (LMUL) from mf8 to m8. VLEN is configurable from 64 to 65536 bits. Additionally, it integrates TestFloat3 for floating-point verification and supports a range of sub-extensions including Zvfh, Zvbb, Zvbc, Zvkg, Zvkned, Zvknha, Zvksed, Zvksh, Zvfbfmin, and Zvfbfwma. The generator leverages the Spike simulator as a golden reference by adding a custom instruction that produces expected results for each test instruction, enabling automatic test generation across the entire RVV instruction set. Users write a simple configuration file per instruction, and the generator produces tests for all instructions almost automatically.

## Key Claims

- Generates unit tests for RVV 1.0 instructions automatically using a Spike-based reference method.
- Supports RV32 and RV64, SEW e8 to e64, LMUL mf8 to m8, and VLEN from 64 to 65536.
- Includes integration with TestFloat3 for floating-point test generation.
- Supports sub-extensions: Zvfh, Zvbb, Zvbc, Zvkg, Zvkned, Zvknha, Zvksed, Zvksh, Zvfbfmin, Zvfbfwma.
- Produces self-verifying binaries suitable for both bare-metal and user-mode environments.
- Known limitations: per-instruction test style only, no tail/mask agnostic support, no fault-only-first tests, no vstart testing, no register group overlap testing, and no coverage guarantees.
- Prerequisites: riscv64-unknown-elf-gcc with RVV 1.0 support, Spike simulator, Go 1.19+, and optionally riscv-pk for user-mode binaries.
- Tests are generated by writing a configuration file per instruction; the tool uses Spike to compute expected results.
- Output binaries are placed in `out/v[vlen]x[xlen][mode]/bin/stage2/`.
- A Nix derivation is also provided for packaging.
